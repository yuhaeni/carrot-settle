## 1. 배경 및 문제 정의

### 1.1 왜 이 시스템을 만드는가?

커머스 플랫폼에서 **주문 → 구매 확정 → 수수료 계산 → 정산 집계 → 지급**까지 이어지는 정산 흐름은 핀테크/커머스 백엔드의 핵심 도메인이다. 그러나 이 전체 흐름을 실무 레퍼런스 수준으로 공개한 오픈소스나 학습 자료는 극히 드물다.

### 1.2 문제 정의

| # | 문제 | 영향 |
|---|------|------|
| P1 | 정산 시스템의 전체 흐름(주문→정산→지급)을 코드 레벨에서 참고할 레퍼런스가 부재 | 커머스 백엔드 개발자가 정산 시스템을 처음 설계할 때 어디서부터 시작할지 막막 |
| P2 | 금액 계산 시 부동소수점 오차로 인한 정산금 불일치 | 주문 1건당 1원 오차 → 월 집계 시 수천~수만 원 차이 → 셀러 클레임 |
| P3 | 동일 정산건 중복 처리 방지 패턴(멱등성, 동시성 제어)의 실무 구현 레퍼런스 부족 | 중복 지급 사고 위험 |
| P4 | 대용량 정산 배치 처리 시 성능 병목 및 OOM 리스크 | 데이터 증가 시 배치 처리 시간 급증, 서비스 안정성 저하 |

### 1.3 해결 방향

입점 셀러가 있는 커머스 플랫폼의 정산 시스템을 MVP 수준으로 직접 설계·구현하여, **주문부터 지급까지의 전체 흐름이 동작하는 실무 레퍼런스**를 만든다.

## 2. 타겟 유저 및 페르소나

### 페르소나 1: 박정산 — 커머스 플랫폼 백엔드 개발자 (시스템을 만드는 사람)

| 항목 | 내용 |
|------|------|
| 나이/경력 | 32세 / 3년차 |
| 소속 | 입점 셀러 50~100개 규모 종합몰 운영사 |
| 기술 스택 | Java, Spring Boot, MyBatis, MySQL |
| 상황 | 주문/결제는 다뤄봤으나, 정산 시스템은 처음 맡게 됨 |

**Pain Points**
1. 구매 확정 → 수수료 → 집계 → 지급의 전체 흐름을 모르니 어디서부터 설계할지 막막
2. float로 수수료를 계산했다가 1원 차이로 셀러 클레임 발생. BigDecimal 반올림 정책을 어떻게 잡아야 하는지 모름
3. 같은 정산건이 2번 처리되거나, 지급이 2번 나가는 상황의 방지 방법에 대한 레퍼런스가 없음
4. `@Scheduled`로 배치를 처리하고 있는데, 데이터 증가 시 처리 시간이 급증하고 Chunk 설계를 모름

**기대 가치**: 주문→지급 전체 흐름을 코드 레벨에서 참고하고, BigDecimal/Strategy 패턴/상태 머신/Spring Batch Chunk 방식의 구현 레퍼런스를 확보

### 페르소나 2: 김셀러 — 소규모 온라인 셀러 (시스템을 사용하는 사람)

| 항목 | 내용 |
|------|------|
| 나이 | 38세 |
| 직업 | 생활용품 온라인 셀러 (플랫폼 3곳 입점) |
| 월 매출 | 약 2,000~3,000만원 |
| 상황 | 플랫폼마다 정산 주기·수수료율이 달라 정산금 예측 불가, 매달 엑셀 수동 대조 |

**Pain Points**
1. PG 수수료 + 플랫폼 수수료 + 프로모션 할인이 복잡하게 얽혀 실제 입금액 예측 불가
2. 플랫폼이 차감한 수수료가 맞는지 직접 계산해볼 방법이 없음
3. 특정 주문의 정산금이 언제, 얼마나 입금되는지 추적이 어려움

**기대 가치**: PG/플랫폼 수수료 분리 조회, 정산 상태 실시간 추적(대기→확정→지급완료), 기간별·상태별 필터링

## 3. 제품 목표 및 성공 지표

### 3.1 제품 목표

> 주문 생성 → 구매 확정 → 수수료 계산 → 정산 배치 집계 → 정산 내역 조회의 **전체 정산 흐름이 End-to-End로 동작**하는 MVP를 6주 내에 완성한다.

## 4. MVP 기능 명세

### 4.0 도메인 모델 개요

```
셀러(Seller)
  └── 상품(Product)
        └── 주문상품(OrderItem)
              └── 주문(Order) ──── 결제(Payment)
                                     └── 정산(Settlement)
                                           └── 지급(Payout)
```

**수수료 구조**
```
PG 수수료      : 결제 금액의 3% (고정)
플랫폼 수수료  : 셀러 등급별 — 일반 5% / 우수 3% / VIP 1%

예시: 주문 10,000원 + 일반 셀러
  PG 수수료     : 300원
  플랫폼 수수료 : 500원
  정산금        : 9,200원
```

**주문 상태 전이**
```
CREATED → PAID → CONFIRMED → SETTLED
                → REFUNDED
```

### 4.1 기능 1: 주문 생성

**User Story**
> 셀러의 상품을 구매하는 고객으로서, 상품을 선택하여 주문을 생성할 수 있다. 이를 통해 정산 프로세스의 시작점이 되는 주문 데이터가 생성된다.

**API**: `POST /api/v1/orders`

**Acceptance Criteria**

| # | Given-When-Then |
|---|-----------------|
| AC-1 | **Given** 유효한 상품 ID와 수량이 포함된 주문 요청이 주어졌을 때, **When** `POST /orders`를 호출하면, **Then** 주문이 `CREATED` 상태로 생성되고, 주문 ID가 포함된 201 응답이 반환된다. |
| AC-2 | **Given** 존재하지 않는 상품 ID로 주문 요청이 주어졌을 때, **When** `POST /orders`를 호출하면, **Then** 400 에러 응답이 반환되고, 주문은 생성되지 않는다. |
| AC-3 | **Given** 여러 상품이 포함된 주문 요청이 주어졌을 때, **When** `POST /orders`를 호출하면, **Then** 주문(Order)과 주문상품(OrderItem)이 하나의 트랜잭션으로 함께 저장되고, 각 주문상품의 금액 합계가 주문 총 금액과 일치한다. |
| AC-4 | **Given** 주문 생성 요청이 주어졌을 때, **When** 주문이 성공적으로 생성되면, **Then** 주문의 생성일시(createdAt)가 기록되고, Swagger UI에서 API 테스트가 가능하다. |

**기술 포인트**
- JPA 엔티티 설계: Order ↔ OrderItem 연관관계 매핑 (`@ManyToOne`, `@OneToMany`)
- BaseEntity 상속 (createdAt, updatedAt 공통 처리)
- 주문 상태 Enum: `@Enumerated(EnumType.STRING)`
- 초기 데이터: Seller, Product를 `data.sql` 또는 `CommandLineRunner`로 세팅

### 4.2 기능 2: 구매 확정

**User Story**
> 플랫폼 운영자로서, 배송 완료 후 환불 기간(7일)이 경과한 주문을 자동으로 구매 확정 처리할 수 있다. 이를 통해 정산 대상 데이터가 확정된다.

**API**: `PATCH /api/v1/orders/{id}/confirm` (수동), `@Scheduled` (자동 — 매일 새벽 2시)

**Acceptance Criteria**

| # | Given-When-Then |
|---|-----------------|
| AC-1 | **Given** 결제 완료(PAID) 상태이고 결제일로부터 7일이 경과한 주문이 존재할 때, **When** 자동 구매 확정 스케줄러가 실행되면, **Then** 해당 주문의 상태가 `CONFIRMED`로 변경되고, 정산 대상(Settlement) 레코드가 함께 생성된다. |
| AC-2 | **Given** `CREATED` 상태(미결제)의 주문이 주어졌을 때, **When** 구매 확정 API를 호출하면, **Then** 상태 전이가 거부되고 400 에러 응답이 반환된다. (허용 전이: PAID → CONFIRMED만 가능) |
| AC-3 | **Given** 이미 `CONFIRMED` 상태인 주문이 주어졌을 때, **When** 구매 확정을 재시도하면, **Then** 중복 처리되지 않고, 기존 상태가 유지된다. |
| AC-4 | **Given** `PAID` 상태의 주문이 주어졌을 때, **When** 환불 요청이 들어오면, **Then** 주문 상태가 `REFUNDED`로 변경되고, 정산 대상에서 제외된다. |

**기술 포인트**
- 상태 전이 검증: Enum에 허용 상태 목록 정의, 잘못된 전이 시 예외 발생
- `@Scheduled(cron = "0 0 2 * * *")`: 매일 새벽 2시 실행
- `@Transactional`: 주문 상태 변경 + 정산 대상 insert를 하나의 트랜잭션으로 묶음

### 4.3 기능 3: 수수료 계산

**User Story**
> 셀러로서, 내 주문에 대한 PG 수수료와 플랫폼 수수료가 각각 얼마인지 투명하게 확인할 수 있다. 이를 통해 실제 정산금을 정확히 예측할 수 있다.

**Acceptance Criteria**

| # | Given-When-Then |
|---|-----------------|
| AC-1 | **Given** 결제 금액이 10,000원이고 셀러 등급이 '일반(STANDARD)'일 때, **When** 수수료 계산이 실행되면, **Then** PG 수수료 300원(3%), 플랫폼 수수료 500원(5%), 정산금 9,200원이 BigDecimal로 정확히 계산된다. |
| AC-2 | **Given** 결제 금액이 999원(원 단위 미만 소수점 발생 가능)일 때, **When** 수수료 계산이 실행되면, **Then** `RoundingMode.HALF_UP` 기준으로 반올림 처리되고, 단건 합산 결과와 일괄 계산 결과가 일치한다. |
| AC-3 | **Given** 셀러 등급이 '우수(PREMIUM)'에서 'VIP'로 변경되었을 때, **When** 새 주문에 대해 수수료 계산이 실행되면, **Then** VIP 등급 수수료율(1%)이 적용되고, 기존 등급 주문의 수수료는 변경되지 않는다. |

**기술 포인트**
- `BigDecimal` 전용 유틸 클래스: RoundingMode, scale 통일 관리
- Strategy 패턴: `FeeCalculationStrategy` 인터페이스 → `PgFeeStrategy`, `PlatformFeeStrategy` 구현체
- `@Embeddable FeeDetail`: PG 수수료, 플랫폼 수수료, 총 수수료를 Value Object로 묶음

### 4.4 기능 4: 정산 배치 집계

**User Story**
> 플랫폼 운영자로서, 구매 확정된 주문들을 일별/주별로 집계하여 셀러별 정산금을 확정할 수 있다. 이를 통해 대량의 주문 데이터를 안정적으로 정산 처리할 수 있다.

**API**: `POST /api/v1/settlements/calculate` (배치 트리거)

**Acceptance Criteria**

| # | Given-When-Then |
|---|-----------------|
| AC-1 | **Given** 구매 확정(CONFIRMED) 상태인 주문 10,000건이 존재할 때, **When** 정산 배치가 실행되면, **Then** 셀러별로 수수료가 차감된 정산금이 정산 테이블에 적재되고, 모든 건의 정산금 합계가 정확하다. |
| AC-2 | **Given** 정산 기준일이 '2026-03-27'로 지정되었을 때, **When** 배치가 실행되면, **Then** 해당 날짜 자정(00:00) 이전에 구매 확정된 주문만 정산 대상에 포함되고, 이후 확정 건은 제외된다. |
| AC-3 | **Given** 배치 실행 중 500번째 chunk에서 오류가 발생했을 때, **When** 동일 JobParameter로 배치를 재실행하면, **Then** 실패 지점(500번째 chunk)부터 처리가 재개되고, 이전에 커밋된 499개 chunk 데이터는 중복 적재되지 않는다. |
| AC-4 | **Given** chunk size가 100, 500, 1000으로 각각 설정되었을 때, **When** 동일 데이터셋으로 배치를 실행하면, **Then** 각 설정별 처리 시간과 메모리 사용량이 측정되어 비교 가능하다. |

**기술 포인트**
- Spring Batch Chunk 방식: `JpaPagingItemReader` → `ItemProcessor`(수수료 차감) → `ItemWriter`(정산 테이블 적재)
- `EntityManager.clear()`: chunk 처리 후 영속성 컨텍스트 초기화 (OOM 방지)
- `JobParameter`: 정산 기준일을 파라미터로 받아 특정 날짜 재정산 지원
- chunk size 튜닝: 100/500/1000 비교 측정 → 최적값 도출

### 4.5 기능 5: 정산 내역 조회

**User Story**
> 셀러로서, 내 정산 내역을 셀러별·기간별·상태별로 조회할 수 있다. 이를 통해 수수료 상세 내역을 확인하고, 예상 입금액을 파악할 수 있다.

**API**: `GET /api/v1/settlements?sellerId={id}&startDate={date}&endDate={date}&status={status}&page={n}&size={n}`

**Acceptance Criteria**

| # | Given-When-Then |
|---|-----------------|
| AC-1 | **Given** 셀러 ID, 기간(시작일~종료일), 정산 상태가 검색 조건으로 주어졌을 때, **When** `GET /settlements`를 호출하면, **Then** 조건에 맞는 정산 내역이 페이징 처리되어 반환되고, 각 건에 PG 수수료·플랫폼 수수료·정산금이 분리 표시된다. |
| AC-2 | **Given** 동일한 조회 요청이 첫 번째 호출된 후, **When** 5분 이내에 동일 요청이 다시 호출되면, **Then** Redis 캐시에서 응답이 반환되어 DB 쿼리가 실행되지 않고, 응답 시간이 캐시 미적용 대비 유의미하게 단축된다. |
| AC-3 | **Given** 정산 상태가 변경(예: PENDING → CONFIRMED)된 건이 존재할 때, **When** 상태 변경 직후 동일 셀러의 정산 내역을 조회하면, **Then** 해당 캐시 키가 무효화되어 DB에서 최신 데이터가 조회된다. |
| AC-4 | **Given** `(seller_id, settlement_date, status)` 복합 인덱스가 적용된 상태에서, **When** EXPLAIN으로 쿼리 실행 계획을 확인하면, **Then** Seq Scan이 아닌 Index Scan이 사용되고, 인덱스 적용 전 대비 쿼리 실행 시간이 개선된다. |

**기술 포인트**
- QueryDSL 동적 쿼리: 셀러별 + 기간별 + 상태별 조건 조합
- Redis Cache Aside 패턴: 확정 데이터 TTL 1시간, 진행 중 TTL 5분, 상태 변경 시 캐시 삭제
- 복합 인덱스: `(seller_id, settlement_date, status)` → EXPLAIN으로 전/후 비교

아래 항목은 MVP에서 **의도적으로 제외**한다. 각 항목의 제외 사유를 명시한다.

| 1 | **실제 PG 결제 연동** | 사업자 등록 필요 | MVP에서는 주문 생성 시 PAID로 직접 전환 |
| 2 | **결제 처리 시뮬레이션 (가상 PG)** | 핵심 정산 흐름 완성이 우선 | 2차 (7주차): PaymentGateway 인터페이스 + FakePaymentGateway |
| 3 | **Redis 분산 락 + 멱등성 키** | 핵심 정산 흐름 완성이 우선 | 2차 (8~9주차): Redisson 분산 락 + Redis DECR 재고 차감 |
| 4 | **지급 처리 (상점별 지급 확정)** | 정산 집계까지가 MVP 범위 | 3차 (10주차): 상태 머신 + Spring Retry |
| 5 | **Kafka 이벤트 기반 아키텍처** | 모놀리식으로 먼저 완성 | 3차 (11주차): 동기 → Kafka 비동기 전환 |

| 7 | **로그인/인증/인가** | 정산 도메인 로직에 집중 | 셀러 ID를 API 파라미터로 직접 전달 |
| 8 | **알림 (이메일/SMS/푸시)** | 정산 흐름의 핵심이 아님 | 미정 |
| 9 | **포인트/쿠폰/프로모션 할인** | 수수료 계산 복잡도 증가 방지 | 미정. MVP는 PG + 플랫폼 수수료만 |
| 10 | **배송 관리** | 구매 확정 트리거로만 사용 (7일 경과) | 배송 추적 시스템은 미구현 |
| 11 | **부분 환불** | MVP 복잡도 관리 | 전체 환불(PAID → REFUNDED)만 지원 |

## 8. 개발 마일스톤

| 단계 | 마일스톤 | 목표 | 주요 산출물 |
|------|---------|------|------------|
| **1차 MVP** | M1~M3 (1~6주차) | 정산 흐름 End-to-End 동작 | 주문 생성 → 정산 배치 → 조회 |
| **2차** | M4~M5 (7~9주차) | 결제 시뮬레이션 + 동시성 제어 + 모니터링 | 가상 PG, 락, Actuator/Prometheus/Grafana, k6 부하 테스트 |
| **3차** | M6~M7 (10~12주차) | 지급 처리 + 비동기/이벤트 기반 전환 | 지급 상태 머신, @Async, Kafka, CI/CD |

### M1: 프로젝트 세팅 + 주문 생성 (1~2주차)
- Docker Compose, JPA 엔티티, `POST /orders`
- 최소 완성 기준: 주문 생성 API 동작, Swagger 확인 가능

### M2: 구매 확정 + 수수료 계산 (3~4주차)
- 상태 전이, `@Scheduled`, Strategy 패턴, BigDecimal 유틸
- 최소 완성 기준: 자동 구매 확정 동작, 수수료 단위 테스트 통과

### M3: 정산 배치 + 조회 (5~6주차)
- Spring Batch Job, QueryDSL 동적 쿼리, Redis 캐시, 복합 인덱스
- 최소 완성 기준: 배치 집계 동작, 조건별 조회 동작

### M4: 결제 시뮬레이션 + 모니터링 (7주차)
- 가상 PG(FakePaymentGateway), Actuator + Prometheus + Grafana, Micrometer 커스텀 메트릭
- 최소 완성 기준: 결제 성공/실패/타임아웃 동작, `/actuator/prometheus` 메트릭 수집

### M5: 동시성 제어 + 부하 테스트 (8~9주차)
- Redis 분산 락(Redisson), JPA 낙관적/비관적 락, 멱등성 키, k6 부하 테스트
- 최소 완성 기준: 동시 요청 100건 중복 처리 없음, TPS/P95 수치 도출

### M6: 지급 처리 (10주차)
- 지급 상태 머신, `@Lock(PESSIMISTIC_WRITE)`, Spring Retry
- 최소 완성 기준: 지급 상태 전이 검증, 중복 지급 방지

### M7: 비동기 처리 + Kafka + CI/CD (11~12주차)
- `@Async` + `CompletableFuture`, Kafka 이벤트 기반 분리, GitHub Actions
- 최소 완성 기준: 비동기 응답 시간 before/after 수치, Kafka 이벤트 발행/수신

**지연 시 Fallback 전략**
- JPA 양방향 매핑 삽질 → 단방향만 사용
- QueryDSL 설정 안 됨 → JPQL로 대체
- Redis 캐시 시간 부족 → 인덱스 전/후 수치만이라도 확보
- Strategy 패턴 어려움 → if-else로 먼저 구현 후 리팩토링

**MVP 완료 시점 체크리스트 (M1~M3)**
- [ ] 주문 생성 → 구매 확정 → 수수료 계산 → 배치 집계 → 조회 End-to-End 동작
- [ ] 정산금 오차율 0원 (테스트 검증)
- [ ] chunk size별 배치 처리 시간 비교표
- [ ] Redis 캐시 적용 전/후 응답 속도 비교
- [ ] 인덱스 적용 전/후 쿼리 실행 시간 비교
- [ ] 핵심 비즈니스 로직 테스트 커버리지 80% 이상

**2~3차 완료 시점 체크리스트 (M4~M7)**
- [ ] Prometheus + Grafana 대시보드에서 배치 지표 실시간 확인
- [ ] Micrometer 커스텀 메트릭 before/after 수치 기록
- [ ] k6 부하 테스트 — 주문 생성 TPS / P95 응답 시간 도출
- [ ] 동시 요청 100건 — 낙관적/비관적 락으로 중복 처리 0건 검증
- [ ] @Async 적용 전/후 API 응답 시간 비교 수치 확보
- [ ] JPA N+1 전/후 쿼리 수 비교 (SQL 로그 기준)

## Appendix: 용어 정의

| 용어 | 정의 |
|------|------|
| 정산 (Settlement) | 구매 확정된 주문에 대해 수수료를 차감한 셀러 지급 금액을 확정하는 프로세스 |
| 구매 확정 (Order Confirmation) | 환불 기간(7일) 경과 후 주문이 최종 확정되어 정산 대상에 포함되는 상태 |
| PG 수수료 | Payment Gateway 이용 수수료. 결제 금액의 3% (고정) |
| 플랫폼 수수료 | 커머스 플랫폼이 셀러에게 부과하는 판매 수수료. 등급별 차등 적용 (일반 5% / 우수 3% / VIP 1%) |
| 정산금 | 주문 금액 - PG 수수료 - 플랫폼 수수료 = 셀러에게 지급될 금액 |
| Chunk | Spring Batch에서 데이터를 일정 단위로 묶어 읽기→처리→쓰기를 수행하는 단위 |
| 멱등성 (Idempotency) | 동일 요청을 여러 번 수행해도 결과가 동일한 성질 |
| Cache Aside | 캐시 미스 시 DB 조회 후 캐시에 저장하는 캐싱 전략 |
