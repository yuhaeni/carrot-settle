# 사이드 프로젝트 기획 노트 - 정산 시스템

## 프로젝트 목적

```
현재: 정산 데이터를 "소비하는 쪽" 경험 (스크래핑 → 대출 한도 계산)
목표: 정산 데이터를 "만드는 쪽" 경험 (주문 → 정산 → 지급)
     = 다나와, 토스, 쿠팡 같은 회사에서 하는 그 정산
```

- 실무에서 정산예정금·매출금 데이터를 다뤄본 도메인 이해는 있음
- 하지만 주문 → 정산 → 지급까지의 전체 흐름을 직접 개발해본 적은 없음
- 퇴사 기간 동안 이 경험을 사이드 프로젝트로 만들어 공백기 증명 + 기술 갭 해소

---

## 면접 스토리 라인

> "실무에서는 정산 데이터를 스크래핑하고 대출 한도를 계산하는 쪽을 담당했는데,
> 정산 시스템 자체가 어떻게 동작하는지 궁금해서 직접 설계·구현해봤습니다.
> 실무에서 정산예정금·매출금을 다뤄본 경험이 있어서 도메인 이해는 있었고,
> 이번에는 주문부터 정산금 확정·지급까지의 전체 흐름을 직접 만들어보면서
> 트랜잭션 설계와 동시성 처리를 경험했습니다."

핵심 키워드: **"소비하는 쪽에서 만드는 쪽으로 확장"**

> "실무에서는 셀러 정산 데이터를 스크래핑하고 대출 한도를 계산하는 시스템을 개발했습니다.
> 정산 데이터의 소비자 입장에서 일하면서, 정산 시스템 자체가 어떻게 동작하는지 궁금해져서
> 주문부터 결제, 정산, 지급까지의 전체 흐름을 직접 설계·구현해봤습니다.
> 특히 동시성 제어와 정산 배치 대용량 처리에서 깊이 있는 경험을 할 수 있었습니다."

## 기술 스택 (확정)

```
Language    : Java 21 (Kotlin 아닌 이유: Java 깊이 강화가 우선)
Framework   : Spring Boot 4.0.x
ORM         : JPA + QueryDSL
DB          : PostgreSQL 16
Cache       : Redis 7 (캐시 + 분산 락)
Batch       : Spring Batch
Container   : Docker Compose
CI/CD       : GitHub Actions
Test        : JUnit5 + Testcontainers
Docs        : Swagger (springdoc-openapi)
```

### Java 21을 쓰는 이유
- 최신 LTS 버전 (2023.09 출시)
- Virtual Thread → 정산 배치에서 플랫폼 스레드 대비 성능 비교 가능
- 패턴 매칭 (switch) 정식 지원, record, sealed class 활용
- 면접에서 "Java 버전별 차이 (11 → 17 → 21)" 답변 소재

### PostgreSQL 16을 쓰는 이유
- Java 21과 동일 시기(2023.09) 릴리스된 최신 안정 버전 → 기술 스택 전체가 최신 LTS로 통일
- COPY 성능 최대 300% 향상, 쿼리 병렬 처리 강화 → 대량 정산 배치 데이터 적재에 유리
- `pg_stat_io` 뷰 추가로 I/O 모니터링 강화 → 배치 성능 튜닝 시 EXPLAIN과 함께 활용 가능
- Spring Boot 4.0.x / Spring Data JPA와 완벽 호환
- 실무에서 사용했던 DB를 유지하되 최신 버전으로 → ORM 전환 학습에만 집중 (새 DB 학습 포인트 분산 방지)

### Redis 7을 쓰는 이유
- 현재 최신 안정 메이저 버전 (2022.04 릴리스, 7.2까지 안정화 완료)
- Redis Functions 도입, ACL v2 강화, Multi-part AOF → 운영 안정성 향상
- Spring Data Redis (Spring Boot 4.0.x) 와 완벽 호환
- 향후 2차 개발(8~9주차)에서 Redisson 분산 락 연동 시 Redis 7의 개선된 성능 활용 가능
- Cache Aside 패턴 + TTL 기반 캐시 전략에 Redis 7의 개선된 메모리 효율성 활용

### Kotlin을 쓰지 않는 이유
- 면접에서 Java 깊이 질문에 못 답하면 탈락
- Kotlin 모르면 "입사 후 배우겠다"로 넘어감
- Kotlin은 이미 일기 프로젝트(우사기)로 경험 있어 이력서에 기재 가능
- Java를 깊게 알면 Kotlin은 2~4주면 실무 투입 가능

### 기능별 기술 상세 + 선택 이유 + 면접 예상 질문

---

#### 1. 주문 생성 (상품 선택 → 주문 생성)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| Spring Boot 4.0.x | Spring Framework 7 + Jakarta EE 11 기반. 3.x→4.0 마이그레이션 경험 + Virtual Thread 강화 지원. "Spring Boot 3과 4 차이" 면접 대응 |
| Java 21 | record→조회 DTO에 활용, sealed class→상태별 타입 안전 처리, Virtual Thread→배치에서 활용 |
| Spring Data JPA | 실무 MyBatis와의 차이를 직접 체감. 영속성 컨텍스트, 더티 체킹, N+1 문제 직접 경험 |
| PostgreSQL | 실무에서 써본 DB 유지 → ORM 전환 학습에만 집중. 새 DB까지 배우면 학습 포인트 분산 |

**면접 예상 질문**

> Q: "왜 JPA를 선택했나요?"
> A: "실무에서는 MyBatis로 정산 조회 로직을 구현했는데, 객체 중심 설계와 연관관계 관리에서 JPA의 장단점을 직접 경험해보고 싶었습니다. 정산 도메인은 엔티티 간 관계가 명확해서 JPA 학습에 적합했습니다."

> Q: "MyBatis와 JPA 차이는?"
> A: "MyBatis는 SQL 직접 작성으로 세밀한 쿼리 제어가 가능하지만, 테이블 변경 시 관련 쿼리를 전부 수정해야 합니다. JPA는 객체 중심 설계로 엔티티 변경만으로 대응 가능하지만, N+1 같은 성능 이슈를 이해해야 합니다. 이 프로젝트에서 N+1을 직접 겪고 fetch join과 @BatchSize로 해결했습니다."

> Q: "영속성 컨텍스트가 뭔가요?"
> A: "엔티티를 관리하는 1차 캐시 영역입니다. 같은 트랜잭션 내에서 동일 엔티티를 다시 조회하면 DB가 아닌 1차 캐시에서 반환하고, 커밋 시점에 변경된 필드를 감지(더티 체킹)해서 자동으로 UPDATE 쿼리를 생성합니다."

> Q: "Java 11과 21의 주요 차이는?"
> A: "17에서 record와 sealed class가 추가되어 불변 DTO와 타입 안전한 상태 관리가 가능해졌고, 21에서는 Virtual Thread가 정식 도입되어 경량 동시성 처리가 가능해졌습니다. 이 프로젝트에서 record는 조회 응답 DTO에, Virtual Thread는 배치 처리에 적용했습니다."

#### 2. 결제 처리 (가상 PG 결제 시뮬레이션)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| 인터페이스 + 가상 구현체 (Adapter 패턴) | 실제 PG 연동 불가(사업자 필요). PaymentGateway 인터페이스 → FakePaymentGateway 구현. 나중에 실제 PG로 교체 시 구현체만 변경 → OCP 준수 |
| @Transactional | 주문 상태 변경(CREATED→PAID) + 결제 기록 저장이 하나의 트랜잭션. 실패 시 롤백 |
| 타임아웃/실패 처리 | 외부 API 호출 시 응답 없으면? → 결제 상태 UNKNOWN → 별도 상태 조회로 확인. 이 "모호한 상태" 처리가 실무 핵심 |
| RestTemplate / WebClient | 가상 PG라도 HTTP 호출 구조를 만들어둠. 동기 vs 비동기 차이 면접 대응 |

**면접 예상 질문**

> Q: "PG 연동을 어떻게 설계했나요?"
> A: "PaymentGateway 인터페이스에 approve(), cancel(), status() 메서드를 정의하고 FakePaymentGateway로 구현했습니다. 실제 PG로 교체 시 구현체만 바꾸면 되는 구조입니다. 실무에서 KG이니시스, NHN KCP 등 여러 PG사와 연동한 경험이 있어서, 다중 PG 대응이 가능한 추상화 구조를 설계했습니다."

> Q: "결제 중 타임아웃이 발생하면 어떻게 처리하나요?"
> A: "타임아웃 발생 시 결제 완료 여부를 알 수 없는 모호한 상태가 됩니다. 결제 상태를 UNKNOWN으로 기록하고, 별도 상태 조회 API를 호출해서 실제 결제 여부를 확인합니다. 결과에 따라 주문 상태를 PAID 또는 FAILED로 갱신하는 보상 로직을 구현했습니다."

> Q: "RestTemplate과 WebClient 차이는?"
> A: "RestTemplate은 동기 블로킹으로 요청당 스레드를 점유합니다. WebClient는 비동기 논블로킹으로 스레드를 효율적으로 사용합니다. 결제처럼 외부 I/O 대기가 긴 경우 WebClient가 유리하지만, 단순 구조를 위해 RestTemplate을 사용하고 필요 시 전환 가능한 구조로 설계했습니다."

#### 3. 동시성 제어 (중복 결제 방지 / 선착순 재고 차감)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| Redis 분산 락 (Redisson) | 서버가 여러 대면 JVM 레벨 synchronized는 무의미. Redis 기반으로 서버 간 동기화. DB 커넥션 점유 없이 락 가능 |
| DB 비관적 락 (@Lock PESSIMISTIC_WRITE) | Redis 장애 시 fallback 또는 단순 구조 대안. Redis 락 vs DB 락 트레이드오프를 아는 것 자체가 어필 |
| 멱등성 키 (Idempotency Key) | 네트워크 장애로 결제 API 중복 호출 가능. UUID 기반 요청 식별 → 이미 처리된 요청이면 기존 결과 반환 |
| Redis DECR (선착순 재고) | 재고 100개에 200명 동시 주문 시? Redis DECR은 원자적(atomic) → DB UPDATE보다 빠르고 안전 |

**면접 예상 질문**

> Q: "동시성 문제를 어떻게 해결했나요?"
> A: "두 가지 레벨로 처리했습니다. 동일 주문 중복 결제 방지는 Redis 분산 락(Redisson)으로 주문 ID를 키로 락을 잡고, 재고 차감은 Redis DECR 명령의 원자성을 활용했습니다."

> Q: "Redis 분산 락과 DB 락의 차이는?"
> A: "DB 비관적 락은 select for update로 row를 잠그는데 DB 커넥션을 점유합니다. 동시 요청이 많으면 커넥션 풀 고갈 위험이 있습니다. Redis 분산 락은 DB 커넥션과 무관하게 관리되고, TTL로 데드락도 방지됩니다. 다만 Redis 장애 시 락이 풀리는 위험이 있어 중요도에 따라 선택합니다."

> Q: "Redis 분산 락에서 데드락 방지는?"
> A: "Redisson의 tryLock에 leaseTime(만료 시간)을 설정합니다. 락을 잡은 프로세스가 장애로 해제하지 못해도 자동 만료됩니다. waitTime으로 락 획득 최대 대기 시간도 제한합니다."

> Q: "synchronized로는 안 되나요?"
> A: "단일 JVM에서는 가능하지만, 서버 2대 이상이면 각 JVM의 synchronized는 독립적이라 동시성을 보장할 수 없습니다. 운영 환경은 다중 서버가 기본이므로 Redis 같은 외부 저장소 기반 분산 락이 필요합니다."

#### 4. 구매 확정 (환불 기간 경과 처리)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| JPA @Enumerated(EnumType.STRING) | ORDINAL(숫자)은 Enum 순서 변경 시 기존 데이터 매핑이 깨짐 → 장애. STRING은 문자열 저장으로 안전 |
| @Transactional | 구매 확정 시 주문 상태 변경 + 정산 대상 insert가 하나의 트랜잭션. 프록시 기반이라 self-invocation 주의 |
| @Scheduled | "매일 새벽 2시에 7일 지난 주문 자동 확정". Spring Batch까지 쓸 필요 없는 단순 반복에 적합 |

**@Scheduled vs Spring Batch (면접 단골)**

```
@Scheduled: 단순 반복 작업 (구매 확정 체크)
            → 실패 시 재시작/건별 추적 불가

Spring Batch: 대량 데이터 처리 (정산 집계)
              → chunk 단위 커밋, 실패 지점부터 재시작
              → Job 실행 이력 관리
```

**면접 예상 질문**

> Q: "@Transactional 동작 원리를 설명해주세요"
> A: "Spring AOP 기반 프록시로 동작합니다. 메서드 호출 시 프록시가 트랜잭션을 시작, 정상 완료 시 커밋, 예외 시 롤백합니다. self-invocation에서는 프록시를 타지 않아 미적용되고, 기본적으로 Unchecked Exception에서만 롤백되므로 필요 시 rollbackFor를 지정해야 합니다."

> Q: "왜 @Scheduled를 쓰고 Batch는 안 썼나요?"
> A: "구매 확정은 '7일 지난 주문 상태 변경'이라는 단순 작업이라 @Scheduled로 충분했습니다. 정산 집계(6번)는 수만 건을 읽고 수수료 계산 후 적재하는 대량 처리라 chunk 커밋과 재시작이 가능한 Spring Batch를 사용했습니다."

> Q: "EnumType.ORDINAL과 STRING 차이는?"
> A: "ORDINAL은 순서(0,1,2)를 저장해서 Enum 순서 변경 시 데이터 매핑이 깨집니다. STRING은 'PAID','CONFIRMED' 문자열을 저장해서 안전합니다. 정산처럼 상태가 핵심인 도메인에서는 반드시 STRING을 써야 합니다."

#### 5. 수수료 계산 (PG + 플랫폼 수수료)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| BigDecimal | float/double은 부동소수점 오차 (0.1+0.2=0.30000000000000004). 정산금 1원 오차 = 장애. RoundingMode 설정 필수 |
| Strategy 패턴 | PG 수수료(3% 고정)와 플랫폼 수수료(등급별 5%/3%/1%) 계산 방식이 다름. 실무 대출 한도 계산에서 동일 패턴 적용한 경험과 연결 |
| @Embeddable (Value Object) | 수수료 내역(PG/플랫폼/총 수수료)을 하나의 값 객체로 묶음. DDD의 Value Object 개념 → "DDD 아시나요?" 대응 |

**수수료 구조 (단순하게)**

```
PG 수수료: 결제 금액의 3% (고정)
플랫폼 수수료: 셀러 등급별
  - 일반: 5%  / 우수: 3%  / VIP: 1%

예시: 주문 10,000원 + 일반 셀러
  PG 수수료: 300원
  플랫폼 수수료: 500원
  정산금: 9,200원
```

**면접 예상 질문**

> Q: "왜 BigDecimal을 사용했나요?"
> A: "금융 도메인에서 float/double은 부동소수점 한계로 정밀 계산이 불가능합니다. 0.1+0.2가 0.3이 아닌 0.30000000000000004가 됩니다. 정산금 1원 오차도 장애이므로 BigDecimal을 사용했고, RoundingMode.HALF_UP으로 원 단위 미만을 처리했습니다."

> Q: "Strategy 패턴을 또 사용하셨네요. 다른 패턴은 고려 안 했나요?"
> A: "Template Method도 고려했지만, 수수료 계산은 상속 계층이 깊어질 필요 없이 계산 로직만 다른 경우라 Strategy가 적합했습니다. 실무에서 대출 한도 계산에 동일 패턴을 적용한 경험이 있어, 같은 판단을 했습니다."

> Q: "@Embeddable이 뭔가요?"
> A: "JPA 임베디드 타입으로, 값 객체를 별도 테이블 없이 소유 엔티티의 컬럼으로 포함시킵니다. 수수료 내역처럼 항상 함께 다니는 데이터를 하나의 객체로 묶어 응집도를 높였습니다."

> Q: "수수료 정책이 바뀌면 어떻게 대응하나요?"
> A: "새 수수료 타입 추가 시 Strategy 구현체 1개만 추가하면 됩니다. 기존 코드 수정 없이 확장 가능한 OCP 구조입니다."

#### 6. 정산 배치 (일별/주별 정산 집계)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| Spring Batch (Chunk) | 대량 데이터 읽기→가공→저장의 전형적인 배치. chunk 단위 커밋으로 중간 실패 시 해당 chunk부터 재처리 가능. 실무 경험 있으니 성능 튜닝에 집중 |
| chunk size 튜닝 | 작으면(100) 커밋 과다, 크면(1000) OOM 위험. size별 처리 시간 측정 → before/after 수치 = 면접 소재 |
| JobParameter | 정산 기준일을 파라미터로 받아서 특정 날짜 재정산 가능. 운영 관점 설계 포인트 |
| Virtual Thread (Java 21) | Batch TaskExecutor를 Virtual Thread로 교체 → 플랫폼 스레드 대비 처리량 비교. "Java 21 신기능을 실제 적용" 어필 |

**면접 예상 질문**

> Q: "Spring Batch를 왜 사용했나요?"
> A: "정산 집계는 수만 건의 구매 확정 데이터를 읽어 수수료 계산 후 적재하는 대량 처리입니다. chunk 단위 커밋으로 중간 실패 시 재시작 가능하고, Job 실행 이력이 메타 테이블에 관리되어 운영 모니터링이 용이합니다."

> Q: "chunk size는 어떤 기준으로 정했나요?"
> A: "100, 500, 1000으로 각각 테스트해서 처리 시간과 메모리를 측정했습니다. 100은 커밋 과다로 I/O 오버헤드, 1000은 메모리 높음. 500에서 균형이 좋아 500으로 설정했습니다."

> Q: "Tasklet과 Chunk 방식의 차이는?"
> A: "Tasklet은 단일 작업을 한 번에 처리, 단순 작업에 적합합니다. Chunk는 Reader→Processor→Writer로 chunk 단위 처리, 대량 데이터에 적합합니다. 정산 집계는 수만 건이므로 Chunk를 선택했습니다."

> Q: "배치 실패 시 어떻게 대응하나요?"
> A: "Job 상태가 FAILED로 기록되면 같은 JobParameter로 재실행 시 실패 지점부터 재시작됩니다. 정산 기준일을 JobParameter로 받아서 특정 날짜만 재실행할 수 있습니다."

> Q: "Virtual Thread를 적용해보셨다고요?"
> A: "Batch TaskExecutor를 Virtual Thread로 교체해서 플랫폼 스레드 대비 처리량을 비교했습니다. DB 조회/적재 같은 I/O 바운드 작업이 많은 배치에서 경량 스레드 전환의 효과를 확인했습니다."

#### 7. 지급 처리 (상점별 지급 확정)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| JPA 비관적 락 (@Lock PESSIMISTIC_WRITE) | 같은 정산건 지급 2번 = 돈 2배. select for update로 row 잠금. 낙관적 vs 비관적 락 차이 설명이 면접 포인트 |
| 상태 머신 설계 | PENDING→CONFIRMED→PAID_OUT/FAILED. 잘못된 전이 방지를 코드로 강제. 단순 if-else가 아닌 구조적 설계 |
| 멱등성 (Idempotency Key) | 지급 API 중복 호출 시에도 1번만 처리. UUID 기반 요청 식별. 결제(3번)와 지급(7번) 모두 필요하다는 이해 |
| Spring Retry + 수동 재처리 API | 일시적 장애: 자동 최대 3회 exponential backoff. 영구 실패: 운영팀 수동 재시도. "실패 시 대응" = 시니어 면접 질문 |

**면접 예상 질문**

> Q: "낙관적 락과 비관적 락의 차이는?"
> A: "낙관적 락은 @Version으로 커밋 시점에 충돌을 감지, 충돌이 드문 경우에 적합합니다. 비관적 락은 조회 시점에 row를 잠그는 방식으로, 충돌이 잦거나 정합성이 절대적으로 중요한 경우에 적합합니다. 정산 지급은 중복 시 금전 피해이므로 비관적 락을 선택했습니다."

> Q: "상태 전이를 어떻게 관리했나요?"
> A: "각 상태에서 전이 가능한 상태 목록을 Enum에 정의했습니다. 상태 변경 시 허용 여부를 먼저 검증하고, 허용되지 않으면 예외를 발생시킵니다. 새 상태 추가 시 누락 위험이 없는 구조입니다."

> Q: "지급 실패 시 어떻게 대응하나요?"
> A: "Spring Retry로 일시적 장애에 최대 3회 exponential backoff 자동 재시도, 그래도 실패하면 FAILED 상태로 기록합니다. 운영팀은 수동 재처리 API로 확인 후 직접 재시도 가능합니다."

> Q: "3번(결제)과 7번(지급) 모두 동시성 처리를 했는데, 차이는?"
> A: "결제 단계는 사용자 동시 요청이 빈번하므로 Redis 분산 락으로 빠르게 제어, 지급 단계는 배치/내부 호출이므로 DB 비관적 락으로 데이터 레벨에서 보호했습니다. 트래픽 특성에 맞는 락 전략을 선택한 것입니다."

#### 8. 정산 내역 조회 (상점별, 기간별)

**기술 + 선택 이유**

| 기술 | 왜 이 기술인가 |
|------|---------------|
| QueryDSL | 셀러별+기간별+상태별 다양한 조건 조합의 동적 쿼리 필요. JPQL은 문자열 기반으로 동적 처리 어려움, Criteria는 가독성 떨어짐. QueryDSL은 타입 안전 + 가독성 |
| Redis (Cache Aside) | 확정된 정산 데이터는 변경 거의 없음 → 캐시 적합. 캐시 있으면 반환, 없으면 DB 조회 후 저장. TTL: 확정 데이터 1시간, 진행 중 5분. 상태 변경 시 캐시 삭제 |
| EXPLAIN + 인덱스 | (seller_id, settlement_date, status) 복합 인덱스. Seq Scan→Index Scan 전환. **before/after 수치 = 이력서 + 블로그 소재** |

**면접 예상 질문**

> Q: "QueryDSL을 왜 선택했나요?"
> A: "정산 내역 조회는 셀러별, 기간별, 상태별 등 조건이 다양하게 조합됩니다. JPQL은 문자열 기반이라 동적 조건 처리가 어렵고, Criteria는 가독성이 떨어집니다. QueryDSL은 자바 코드로 타입 안전하게 작성 가능하고, 컴파일 타임 오류 검출이 장점이었습니다."

> Q: "캐시 전략을 설명해주세요"
> A: "Cache Aside 패턴을 적용했습니다. 조회 시 Redis에 있으면 즉시 반환, 없으면 DB 조회 후 저장합니다. 확정 데이터는 TTL 1시간, 진행 중은 5분으로 설정하고, 상태 변경 시점에 캐시 키를 삭제해 정합성을 유지했습니다."

> Q: "Write Through, Write Behind, Cache Aside 차이는?"
> A: "Write Through는 DB 쓰기 시 캐시도 동시 갱신, 정합성 좋지만 쓰기 느림. Write Behind는 캐시에 먼저 쓰고 나중에 DB 반영, 성능 좋지만 유실 위험. Cache Aside는 캐시 미스 시에만 DB→캐시 저장, 읽기 위주에 적합. 정산 조회는 읽기가 대부분이라 Cache Aside를 선택했습니다."

> Q: "쿼리 튜닝 경험을 설명해주세요"
> A: "셀러ID+정산일자 조회 패턴이 빈번해서 EXPLAIN으로 분석했습니다. 인덱스 없이 Seq Scan 발생 확인 후, (seller_id, settlement_date, status) 복합 인덱스를 추가해 Index Scan으로 전환했습니다. 쿼리 실행 시간 XXms→XXms 개선."

> Q: "캐시와 DB 정합성은 어떻게 보장하나요?"
> A: "정산 상태 변경 시점(구매 확정, 집계, 지급 완료)에 해당 캐시 키를 삭제합니다. Write Through 대신 Cache Aside를 선택한 이유는, 정산 데이터는 조회가 대부분이고 변경은 배치나 상태 전이 시에만 발생하기 때문입니다."

### 1차 개발 (MVP) — 6주

> 목표: "주문 → 구매확정 → 수수료 계산 → 정산 배치 → 조회"가 동작하는 정산 시스템

#### 1~2주차: 프로젝트 세팅 + 주문 생성 (기능 1번)

```
[1주차 전반] 환경 구성
├── Spring Boot 4.0.x + Java 21 + Gradle 프로젝트 생성
├── PostgreSQL + Redis → Docker Compose로 구성
├── Spring Data JPA + QueryDSL 의존성 설정
├── Swagger (springdoc-openapi) 설정
└── GitHub 저장소 생성 + .gitignore 설정

[1주차 후반] 엔티티 설계
├── ERD 상세 설계 (셀러, 상품, 주문, 주문상품, 정산, 지급)
├── JPA 엔티티 클래스 작성
├── 연관관계 매핑 (@ManyToOne, @OneToMany)
├── BaseEntity (생성일시, 수정일시) 공통 처리
└── 주문 상태 Enum 정의 (CREATED → PAID → CONFIRMED → ...)

[2주차 전반] 주문 생성 API
├── POST /orders 주문 생성 API
├── 셀러 + 상품 기본 데이터 세팅 (data.sql or CommandLineRunner)
├── 주문 생성 시 주문상품 함께 저장
└── REST API 설계 원칙 적용 (상태코드, 에러 응답 형식)
```

## 타겟 유저 페르소나

### 페르소나 1: 커머스 플랫폼 백엔드 개발자 (시스템을 만드는 사람)

**인구통계학적 정보**

| 항목 | 내용 |
|------|------|
| 이름 | 박정산 |
| 나이 | 32세 |
| 직업 | 중소 커머스 플랫폼 백엔드 개발자 (3년차) |
| 소속 | 입점 셀러 50~100개 규모의 종합몰 운영사 |
| 기술 스택 | Java, Spring Boot, MyBatis, MySQL |

**핵심 특징 및 성격**

- 주문/결제는 다뤄봤지만 **정산 시스템은 처음 맡게 됨**
- "정산 = 매출에서 수수료 빼면 되는 거 아닌가?" 정도의 이해 수준
- 기술적으로 성실하지만, 정산 도메인의 복잡성(상태 관리, 동시성, 멱등성)을 아직 모름
- 정산 시스템 설계에 대한 실무 수준 레퍼런스를 찾기 어려워 답답함

**정산 업무에서 겪는 어려움 (Pain Point)**

1. **정산 흐름을 모름** — 구매 확정→수수료→집계→지급의 전체 흐름을 모르니 어디서부터 설계할지 막막
2. **금액 계산 정합성 불안** — float로 계산했다가 1원 차이로 셀러 클레임. BigDecimal 반올림 정책을 어떻게 잡아야 하는지 모름
3. **동시성/중복 처리 경험 없음** — 같은 정산건이 2번 처리되면? 지급이 2번 나가면? 방지 방법 레퍼런스가 없음
4. **배치 설계 경험 부족** — @Scheduled로 때우고 있는데 데이터가 늘면서 처리 시간 증가. Chunk 설계를 어떻게 해야 할지 모름

**우리 서비스가 주는 가치**

- 주문→구매확정→수수료→정산배치→지급의 **전체 흐름을 코드 레벨에서 참고 가능**
- BigDecimal + Strategy 패턴으로 **수수료 계산 정합성 확보 방법** 학습
- 상태 머신 + 멱등성 + 비관적 락으로 **중복 지급 방지 패턴** 학습
- Spring Batch Chunk 방식의 **실제 정산 배치 구현 레퍼런스**

### 페르소나 2: 소규모 셀러 (시스템을 사용하는 사람)

**인구통계학적 정보**

| 항목 | 내용 |
|------|------|
| 이름 | 김셀러 |
| 나이 | 38세 |
| 직업 | 생활용품 온라인 셀러 (플랫폼 3곳 입점) |
| 월 매출 | 약 2,000~3,000만원 |
| 입점 플랫폼 | 쿠팡, 스마트스토어, 자사몰 |

**핵심 특징 및 성격**

- 플랫폼마다 정산 주기, 수수료율, 정산 기준이 달라서 **정산금 예측이 어려움**
- 매달 엑셀로 플랫폼별 정산 내역을 수동 대조
- "이번 달 실제로 얼마 들어오는 거지?"를 빨리 알고 싶음
- 수수료가 제대로 차감됐는지 검증할 방법이 없어 불안

**정산 업무에서 겪는 어려움 (Pain Point)**

1. **정산금 예측 불가** — 플랫폼 수수료+PG 수수료+프로모션 할인이 복잡하게 얽혀서 실제 입금액을 미리 알 수 없음
2. **플랫폼별 정산 기준 상이** — A 플랫폼: 구매 확정 후 7일, B 플랫폼: 월 2회 정산. 각 플랫폼 정산 상태를 일일이 확인해야 함
3. **수수료 검증 불가** — 플랫폼이 차감한 수수료가 맞는지 직접 계산해볼 방법이 없음
4. **정산 내역 추적 어려움** — 특정 주문의 정산금이 언제, 얼마나 입금되는지 추적 어려움. 환불 건 정산 제외 확인도 수작업

**우리 서비스가 주는 가치**

- 주문별 수수료 상세 내역 **투명하게 조회** (PG 수수료, 플랫폼 수수료 분리)
- 정산 상태 추적 **(대기→확정→지급완료) 실시간 확인**
- 정산 주기별 **예상 입금액 사전 파악**
- 기간별/상태별 정산 내역 필터링으로 **수작업 대조 시간 절감**

### 🔴 리스크 1: 정산 금액 정합성 오류 (가장 치명적)

**문제 예측**
- 반올림 처리가 일관되지 않으면 주문 단위 1원 차이가 월 집계에서 수천 원 오차로 확대
- 예: 999원 × 5% = 49.95원 → HALF_UP이면 50원, HALF_DOWN이면 49원 → 10,000건이면 최대 10,000원 차이

**대응 방안**
1. BigDecimal 유틸 클래스를 하나 만들어 프로젝트 전체에서 공용 사용 (RoundingMode, scale 한 곳에서 관리)
2. 정산 금액 검증 테스트 최우선 작성 (단건 → 다건 합산 → 월 집계, 경계값: 0원, 1원, 소수점)
3. 단건 합산 vs 일괄 계산 결과 비교 테스트

> 면접: "정산 금액 정합성을 보장하기 위해 BigDecimal 연산을 공용 유틸로 통일하고, 단건 합산과 일괄 계산의 결과가 일치하는지 검증하는 테스트를 작성했습니다."

---

### 🔴 리스크 2: 정산 배치와 상태 변경의 동시 실행 충돌

**문제 예측**
- 새벽 2시 배치 실행 중(구매 확정 주문 집계) → 2시 3분 고객 환불 요청 → 주문 REFUNDED
- 배치가 이미 해당 주문을 읽어서 정산에 포함 → 환불 건인데 셀러에게 돈 지급

**대응 방안**
1. 정산 배치 조회 기준을 "기준일 자정(00:00) 이전 구매 확정 건"으로 고정
2. 정산 대상 확정 시 스냅샷 방식 — 금액/수수료를 정산 테이블에 복사 (원본 변경 영향 없음)
3. 이미 정산된 건의 환불 → 기존 정산 수정하지 않고 마이너스 차감 건을 새로 생성

> 면접: "배치 중 환불 발생하는 동시성 이슈를 고려해서, 스냅샷 방식으로 금액을 복사하고, 환불은 별도 차감 건으로 처리하는 설계를 적용했습니다."

---

### 🔴 리스크 3: Spring Batch 대용량 처리 시 OOM

**문제 예측**
- JPA 환경에서 영속성 컨텍스트에 엔티티가 계속 누적 → chunk size 작게 잡아도 1차 캐시 안 비워지면 OOM
- MyBatis에서는 발생하지 않던 문제 → JPA 처음 써보는 상황에서 예측 못할 가능성 높음

**대응 방안**
1. JpaPagingItemReader 사용 (페이지 단위 조회, pageSize = chunk size)
2. chunk 처리 후 EntityManager.clear()로 영속성 컨텍스트 초기화
3. chunk size별 메모리 사용량 모니터링 (VisualVM or Actuator) → 비교표 작성 = 블로그 + 면접 소재

> 면접: "JPA 배치에서 영속성 컨텍스트 누적으로 OOM이 발생할 수 있다는 점을 인지하고, chunk 단위 EntityManager.clear()로 해결했습니다."
